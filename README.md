# Community Forum

A course-scoped discussion forum. Students see and bookmark posts for the courses they're enrolled in; moderators can see every course and remove posts. Built as a pnpm monorepo with a layered Elysia/PostgreSQL API and a Next.js/React Query frontend.

Authentication is intentionally stubbed (see [Demo Users & Authentication](#demo-users--authentication)) — this is a take-home assignment, not a production login system. Authorization, on the other hand, is fully real and enforced server-side. See [`NOTES.md`](./NOTES.md) for the full engineering write-up behind these decisions.

## Features

- **Course feed** — paginated list of posts for a course, ordered newest first.
- **Saved posts (bookmarks)** — save/unsave a post, with idempotent save and soft-delete semantics so nothing can be double-saved or lose its history.
- **Moderator post deletion** — moderators can permanently remove a post; students never see the option and can't do it via the API either.
- **Pagination** — cursor-free page/pageSize pagination with server-side clamping (an out-of-range page returns the last real page instead of erroring or coming back empty).
- **Role-based access** — students only see courses they're enrolled in; moderators see everything.
- **Demo user switcher** — swap between three seeded identities in the UI to see authorization behave differently per role, with no real login.
- **English/Spanish UI** — every string comes from a message catalog (`next-intl`).
- **Responsive layout** — full sidebar on desktop, a collapsed icon rail on tablet, a horizontal tab bar on mobile.

## Tech Stack

| Layer | Choice | Why |
|---|---|---|
| Monorepo | pnpm workspaces | One install, one lint/typecheck/test pass across `web`, `server`, and `shared`, while each stays independently runnable. |
| Backend framework | [Elysia](https://elysiajs.com/) | Small, fast, strong TypeScript inference on route handlers, plugin-based composition (auth, error handling). |
| Database | PostgreSQL | Relational data (users/courses/enrollments/posts/saves) with real foreign keys and constraints doing the integrity work, not just application code. |
| ORM | [Drizzle](https://orm.drizzle.team/) | SQL stays visible (no generated black box); row types are inferred directly from schema definitions. |
| Validation | [Zod](https://zod.dev/) | Every request parameter and every client env var is parsed and rejected early if it doesn't match, both on the server and in `web/src/config/env.ts`. |
| Frontend framework | Next.js (App Router) | Server Components by default, `"use client"` only where there's real interactivity — keeps pages thin. |
| Server state | TanStack Query v5 | Caching, de-duplication, and background refetching for feed/saved-posts data, plus `keepPreviousData` so pagination doesn't flash a skeleton on every page change. |
| Styling | Tailwind CSS | Utility classes plus a small token set (`background`, `foreground`, `muted`, `primary`, …) for consistent theming, including dark mode. |
| Component structure | Atomic Design | Atoms → molecules → organisms → templates, so every new UI piece has an obvious home. |
| i18n | next-intl | Locale-prefixed routing, message catalogs per locale, no hardcoded strings in components. |
| Testing | Vitest | Same test runner across all three workspaces. |

## Prerequisites

- Node.js ≥ 20.18
- pnpm ≥ 9 (`corepack enable` will pick up the pinned version automatically)
- Docker (for local PostgreSQL) — or your own PostgreSQL 16 instance if you'd rather not use Docker

## Getting Started

### 1. Clone the repository

```bash
git clone <repo-url> community-forum
cd community-forum
```

### 2. Install dependencies

Installs `web`, `server`, and `shared` together via pnpm workspaces.

```bash
pnpm install
```

### 3. Configure environment variables

Copies the example env file. The defaults already match the Docker Postgres setup below, so no edits are needed to get started.

```bash
cp .env.example .env
```

### 4. Start PostgreSQL

Runs a local Postgres 16 container in the background.

```bash
docker compose up -d postgres
```

### 5. Apply the database schema

Runs the Drizzle migration that creates the tables.

```bash
pnpm db:migrate
```

### 6. Load demo data

Seeds courses, users, posts, and saved posts.

```bash
pnpm db:seed
```

### 7. Start the app

Runs the web app and API together in watch mode.

```bash
pnpm dev
```

Once running:

- Web app: **http://localhost:3000**
- API: **http://localhost:4000/api**
- Postgres: **localhost:5432**

`pnpm dev` runs the Next.js dev server and the Elysia API in parallel (`tsx watch`), so both restart on file changes.

> **Note:** `pnpm db:seed` fully resets the demo tables (`users`, `courses`, `enrollments`, `posts`, `saved_posts`) inside a transaction before reinserting — it's safe and idempotent to re-run any time you want to get back to a clean demo state (for example, after clicking around and bookmarking/deleting things).

## Demo Users & Authentication

There's no login page. Every API request carries an `x-user-id` and `x-role` header identifying who's making the request (`server/src/api/plugins/AuthPlugin.ts`); missing or malformed headers get a `401`. The web app's "Demo User" switcher (top right) just changes which pair of headers gets attached to outgoing requests — nothing about sessions or cookies is involved.

Three identities are seeded and selectable in the UI:

| Name | Role | User ID | Access |
|---|---|---|---|
| Alice Johnson | student | `10000000-0000-4000-8000-000000000002` | Enrolled in **Biology** only |
| Bob Chen | student | `10000000-0000-4000-8000-000000000003` | Enrolled in **Biology** and **Chemistry** |
| Morgan Lee | moderator | `10000000-0000-4000-8000-000000000001` | Every course, plus post deletion |

You can exercise the API directly the same way the frontend does:

```bash
# Alice's feed for Biology (course id 20000000-0000-4000-8000-000000000001)
curl http://localhost:4000/api/courses/20000000-0000-4000-8000-000000000001/posts \
  -H "x-user-id: 10000000-0000-4000-8000-000000000002" \
  -H "x-role: student"

# Alice trying to view Chemistry (course id ...002) she isn't enrolled in → 403
curl http://localhost:4000/api/courses/20000000-0000-4000-8000-000000000002/posts \
  -H "x-user-id: 10000000-0000-4000-8000-000000000002" \
  -H "x-role: student"

# Moderator deleting a post → 204
curl -X DELETE http://localhost:4000/api/posts/<post-id> \
  -H "x-user-id: 10000000-0000-4000-8000-000000000001" \
  -H "x-role: moderator"
```

## API Reference

All routes are mounted under `/api`. Pagination query params (`page`, `pageSize`, default `1`/`5`, max `pageSize` of `100`) apply to both list endpoints and return:

```json
{
  "items": [ /* ... */ ],
  "page": 1,
  "pageSize": 5,
  "totalItems": 12,
  "totalPages": 3,
  "hasPreviousPage": false,
  "hasNextPage": true
}
```

| Method | Path | Auth | Notes |
|---|---|---|---|
| `GET` | `/courses/:courseId/posts` | any role | 403 if a student isn't enrolled in the course |
| `POST` | `/posts/:postId/save` | any role | Bookmarks a post; idempotent if already saved |
| `DELETE` | `/posts/:postId/save` | any role | Removes a bookmark (soft delete) |
| `GET` | `/me/saved-posts` | any role | Paginated list of the caller's own saved posts |
| `DELETE` | `/posts/:postId` | moderator only | 403 for students, 404 if the post doesn't exist, 204 on success |

Errors come back as `{ "error": { "code": "...", "message": "...", "details": ... } }` with a matching HTTP status (`400` validation, `401` missing/invalid auth headers, `403` forbidden, `404` not found).

## Database

Five tables: `users`, `courses`, `enrollments`, `posts`, `saved_posts` (see `server/src/db/schema`). A few details worth knowing:

- `saved_posts` uses soft delete (`deleted_at`) instead of hard delete, so a save can be reactivated instead of re-inserted.
- A **partial unique index** on `saved_posts (user_id, post_id) WHERE deleted_at IS NULL` guarantees at most one *active* save per user/post at the database level — not just something the application code tries to prevent.
- Seed data: 2 courses (Biology, Chemistry), 3 users (Alice, Bob, Morgan the moderator), 20 posts (12 Biology / 8 Chemistry) with staggered timestamps so pagination has something real to page through, and 11 saved-post rows spread across users so bookmarks visibly differ per demo identity.

## Architecture & Design Decisions

The short version — see [`NOTES.md`](./NOTES.md) for the full reasoning behind each of these:

- **Layering**: `route → service → repository → db`. Routes are thin (auth, validate, call one service method, map the result to a status code). Services hold every business decision. Repositories only run Drizzle queries. This is enforced by convention, not by a lint rule, but it's consistent across the codebase — `posts.routes.ts` is ~40 lines because it isn't doing anything except wiring.
- **No real auth, real authorization**: identity comes from request headers (see above); *what a given role is allowed to do* is checked against real enrollment and role data on every request, never trusted from the client.
- **Optimistic UI, server-confirmed**: bookmark toggles update the UI immediately and roll back on failure, then every mutation invalidates the relevant caches once it settles so the UI reconciles with what the server actually did.
- **Atomic design on the frontend**: `atoms` (Button, Badge, Avatar) → `molecules` (Pagination, EmptyState, ViewToggle) → `organisms` (NavSidebar, AppHeader, MobileNav) → `templates` (AppLayout). New UI has an obvious place to live.
